## 2026-06-08

### 项目初始化
- 确定技术栈：**Flask** + **SQLAlchemy** + **SQLite** + **Jinja2** + **Bootstrap 4**
- 采用蓝图（Blueprint）模块化组织路由：
  - `auth` — 用户认证（注册/登录/登出）
  - `question_bank` — 题库管理（题目CRUD、专业管理）
  - `exam` — 考试管理（创建/发布/结束考试）
  - `execution` — 学生参加考试
  - `result` — 考试成绩与记录
  - `ai_service` — AI 出题 / AI 批改
- 设计并实现核心数据模型（`database/models.py`）：
  - `User` — 用户（老师 / 学生）
  - `Major` — 专业分类
  - `Question` — 题目（含题型、难度、选项JSON、答案、解析）
  - `Exam` — 考试（含时间、时长、状态）
  - `ExamQuestion` — 考试-题目关联（含分值、排序）
  - `ExamStudent` — 被邀请参加考试的学生
  - `ExamSession` — 学生考试会话
  - `Answer` — 学生作答记录
  - `Result` — 考试成绩

### 数据库切换
- 从 PostgreSQL 切换到 **SQLite**
- 原因：缺少 `psycopg2` 驱动，导致 `ModuleNotFoundError`
- 解决：`config.py` 中 `SQLALCHEMY_DATABASE_URI` 改为 `sqlite:///exam_system.db`

### 解决循环导入问题
- **现象**：`ImportError: cannot import name 'User' from partially initialized module`
- **根因**：`app.py` → `database/models.py` → `app.py` 形成循环依赖
- **解决**：新建 `database/__init__.py`，将 `db = SQLAlchemy()` 提取到该文件中
  - `models.py` 改为 `from database import db`
  - `app.py` 中 `load_user` 函数内部延迟导入 `User`

## 2026-06-13

### 注册邮箱唯一约束冲突修复（`auth/routes.py`）
- **现象**：第二个不填邮箱的用户注册时报错 `UNIQUE constraint failed: user.email`
- **根因**：空邮箱存为 `''`（空字符串），SQLite 唯一约束下多个 `''` 会冲突
- **解决**：`email = request.form.get('email', '').strip() or None`，空邮箱存 `None`（SQL NULL），SQLite 允许多个 NULL 共存

### ExamQuestion 模型缺少 relationship（`database/models.py`）
- **现象**：添加题目页面报错 `'ExamQuestion object' has no attribute 'question'`
- **根因**：`ExamQuestion` 只有 `question_id` 外键，未声明 `question` 关系属性
- **解决**：添加 `question = db.relationship('Question')`，模板中 `eq.question.content` 即可正常访问

### SQLAlchemy 2.0 LegacyAPIWarning 修复（`app.py`）
- **现象**：`load_user` 回调触发 `LegacyAPIWarning: The Query.get() method is considered legacy`
- **解决**：`User.query.get(int(user_id))` → `db.session.get(User, int(user_id))`

### 考试时间校验时区不一致修复（`exam_execution/routes.py`）
- **现象**：学生在考试时间内点击"开始"却提示"考试尚未开始"
- **根因**：`datetime.utcnow()` 返回 UTC 时间，而 `exam.start_time` 存的是本地时间，中国 UTC+8 下比较错位
- **解决**：所有时间比较统一使用 `datetime.now()`（本地时间），共修改 4 处：
  - 考试时间校验（`now < exam.start_time`）
  - 创建考试会话的 `start_time`
  - 提交考试的 `end_time`
  - 成绩记录的 `submitted_at`

### SQLite 数据库锁超时修复（`config.py` + `exam_execution/routes.py`）
- **现象**：学生提交考试时报错 `database is locked`
- **根因**：AI 批改需多次 API 调用（约 10 秒），期间持有 SQLite 写事务，而 SQLite 默认锁超时仅 5 秒
- **解决**：
  - `config.py`：增加 `SQLALCHEMY_ENGINE_OPTIONS = {"connect_args": {"timeout": 30}}`，锁超时提升至 30 秒
  - `submit_exam`：确保先 commit 考试会话状态（释放锁），再调用 AI 批改

### AI 出题 `scoped_session` 参数调用修复（`ai_service/services.py`）
- **现象**：AI 出题报错 `'scoped_session' object has no attribute 'session'`
- **根因**：`ai_generate_questions` 和 `ai_grade_exam` 的参数 `db` 实际传入的是 `db.session`（scoped_session），但代码中误用了 `db.session.get()`（双重访问 `.session`）
- **解决**：
  - `ai_generate_questions`：`db.session.get()` → `db.get()`（参数已是 session）
  - `ai_grade_exam`：同上
  - `ai_generate_analysis`：参数为 `None`，改用局部导入 `_db.session.get()`

### AI API 超时与重试机制（`ai_service/client.py`）
- **问题**：生成多道题目时 API 响应超过 30 秒导致 `Read timed out`
- **解决**：
  - 超时时间从 30 秒提升至 60 秒
  - 新增自动重试机制（`max_retries=2`），超时后自动重试一次
  - 非超时类错误（4xx/5xx/连接失败）不重试，直接报错
  - 日志打印每次请求尝试次数，方便排查

### AI 出题枚举值大小写不一致修复（`ai_service/routes.py`）
- **现象**：AI 生成的单选题筛选时选“单选题”查询不到
- **根因**：下拉框 `value` 使用了 `dir()` 返回的属性名（`SINGLE_CHOICE` 大写），而非实际枚举值（`single_choice` 小写），导致存入数据库的题型值与筛选查询不匹配
- **解决**：
  - 下拉框生成：`'value': t` → `'value': getattr(QuestionTypeEnum, t)`
  - 映射字典键从小写改为大写：`'SINGLE_CHOICE': '单选题'` → `'single_choice': '单选题'`
  - 难度映射同步修复：`'EASY': '简单'` → `'easy': '简单'`
- **数据迁移**（`app.py`）：启动时自动将已有题目的大写题型/难度值转为小写

### AI批改性能优化：客观题本地比对（`ai_service/services.py`）
- **原方案**：每道题均调用 AI API 批改，10 道题需 10 次 API 请求，耗时数十秒
- **新方案**：
  - **客观题**（单选/多选/填空/判断）：直接与题库答案本地比对，即时评分
    - 单选题/判断题：忽略大小写后精确匹配
    - 多选题：排序后比较（允许答案顺序不同）
    - 填空题：忽略大小写和多余空格
  - **主观题**（问答/编程/应用/计算）：仍调用 AI 语义批改，失败时给 0 分等人工评分
- **性能提升**：纯客观题考试提交后几乎即时完成，无需等待 API

### 提交/切屏交互优化（`templates/execution/take_exam.html`）
- **防重复提交**：新增 `isSubmitting` 标志，倒计时归零/切屏自动提交/手动点击三处均加锁
- **切屏弹窗与提交遮罩不再重叠**：提交时先 `$('#switchModal').modal('hide')` 关闭切屏弹窗，再显示提交遮罩
- **提交中禁用切屏检测**：`visibilitychange` 监听器在 `isSubmitting=true` 时直接返回，不再触发上报和弹窗
- **遮罩层 pointer-events: none**：阻止点击穿透，防止意外操作
- **遮罩提示简化**：「AI 正在批改中，可能需要数十秒」改为「正在自动批改中」（因为客观题已无需等待 AI）

### 答案解析页正确性判断修复（`result/routes.py` + `templates/result/analysis.html`）
- **根因**：旧版 AI 批改对每道题调 API，API 失败时 `answer.is_correct` 和 `answer.score` 保留为 `None`，模板判断 None 为 falsy 显示「错误」，得分显示「None 分」
- **修复**：`view_analysis` 路由中检测 `is_correct` 是否为 None，若是则按客观题本地比对重新判断：
  - 单选/判断：忽略大小写匹配
  - 多选：排序后比较
  - 填空：忽略大小写和空格
  - 主观题：有分数则视为正确，否则标记为待人工评分
- **模板改进**：显示每道题满分分值、成绩显示 `得分 / 满分`，未作答时显示「（未作答）」

### 仪表板「已结束的考试」成绩显示修复（`exam_execution/routes.py` + `templates/execution/dashboard.html`）
- **根因**：原模板用 `{% for r in current_user.results %}{% set my_result = r %}{% endfor %}` 查找对应 Result，但 Jinja2 内层 `{% set %}` 无法在外层作用域生效，导致 `my_result` 始终为 `None`
- **修复**：路由层构建 `result_map = {exam_id: Result}` 字典传入模板，模板直接用 `result_map.get(exam.id)` 获取 Result
- **显示逻辑优化**：
  - 有成绩且 `total_score > 0`：显示「得分 / 满分」（着色）
  - 有成绩但 `total_score == 0`：显示「待批改」
  - 无 Result 记录：显示「未作答」

### 自动结束考试时正确批改客观题（`app.py`）
- **原方案**：`auto_end_expired_exams` 仅累加 `Answer.score or 0`，但 Answer 的 `score` 字段可能为 None（未批改）
- **新方案**：在创建 Result 前对每道未批改的客观题做本地比对，设置 `is_correct` 和 `score`，主观题标记 `needs_manual_grade=True`
- 确保超时自动结束的考试也能正确计算成绩，不再出现「得分 0 / 总分 0」的空记录
